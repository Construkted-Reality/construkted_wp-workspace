# Fix S3 5GB Upload Limit via Multipart Upload

## Overview

Users uploading files >5GB to Wasabi S3 receive an `EntityTooLarge` error. The current implementation sends the entire file as a single PUT request to a presigned S3 URL. S3 enforces a hard 5GB (5,368,709,120 bytes) limit on single PUT operations. This is an immutable protocol-level constraint — not configurable via bucket policy or IAM.

The fix: for files >5GB, use S3 multipart upload (CreateMultipartUpload → UploadPart × N → CompleteMultipartUpload), which supports objects up to 5TB. Files ≤5GB continue using the existing single PUT path unchanged.

**Scope:** Fix the >5GB limit only. Upload resume capability is deferred. Security hardening of existing endpoints is deferred to a separate PR.

## Problem Statement

- **Error:** `<Code>EntityTooLarge</Code>` with `MaxSizeAllowed: 5368709120`
- **Trigger:** Any file >5GB uploaded via the "Add new asset" page at `construkted.com/add-new-post/`
- **Root cause:** `TSZF_UPloader.ts:205` sends `xhr.send(file.getNative())` as a single PUT to a presigned URL generated by `upload.php:177-232`
- **Affected asset types:** All four — Polygon Mesh, Tiled Model, Point Cloud, Orthomosaic (they share the same upload path)
- **Client-side max:** `frontend-form.ts:79` sets `maxFileSize = 50 * 1024 * 1024` KB (~50GB), so users can select files far exceeding the 5GB S3 limit with no warning
- **Realistic max file size:** 20-30GB typical large uploads, 50GB theoretical ceiling

## Proposed Solution

### Architecture

```
Browser                          WordPress (PHP)                 Wasabi S3
──────                          ───────────────                 ─────────
1. Select file (>5GB)
2. POST /wp-admin/admin-ajax.php ──→
   action=tszf_create_multipart     Authenticate user
                                    SigV4-signed POST ─────────→ CreateMultipartUpload
                                    Store transient:               Returns uploadId
                                      uploadId → {userId, key}
                                ←── { uploadId, key }

3. For each 10MB chunk (sequentially):
   POST /wp-admin/admin-ajax.php ──→
   action=tszf_get_upload_part_url   Validate uploadId ownership
                                     Generate SigV4 presigned URL
                                       for UploadPart
                                ←── { url }
   PUT chunk to presigned URL ─────────────────────────────────→ Returns ETag
   ⚠️ Part 1: verify ETag header
      is readable (CORS check)
   Collect ETag in array

4. POST /wp-admin/admin-ajax.php ──→
   action=tszf_complete_multipart    Validate uploadId ownership
                                     SigV4-signed POST ────────→ CompleteMultipartUpload
                                     Delete transient              Object assembled
                                ←── { success }

5. On explicit cancel:
   POST /wp-admin/admin-ajax.php ──→
   action=tszf_abort_multipart       Validate uploadId ownership
                                     SigV4-signed DELETE ──────→ AbortMultipartUpload
                                     Delete transient              Parts deleted
```

**Files ≤5GB:** Use existing single PUT path. No changes to current code path.

### Threshold Logic

```typescript
const MULTIPART_THRESHOLD = 5 * 1024 * 1024 * 1024; // 5GB

if (fileSize > MULTIPART_THRESHOLD) → multipart upload (new code)
else                                → single PUT (existing code, untouched)
```

**Rationale:** The existing single PUT path works correctly for all files ≤5GB. Routing those files through new multipart code adds regression risk for zero benefit. The multipart path is only for files that currently fail.

### Part Size

```typescript
const PART_SIZE = 10 * 1024 * 1024; // 10MB fixed
```

At 50GB (the realistic ceiling), this produces 5,000 parts — well under S3's 10,000 limit. No dynamic sizing needed.

## Technical Considerations

### Repos Affected

| Repo | Branch From | Files Changed |
|------|-------------|---------------|
| `construkted.uploadjs` | `main` | `src/TSZF_UPloader.ts` |
| `construkted_reality_v1.x` | `develop` | `wp-content/themes/gowatch-child/includes/frontend-submission/class/upload.php` |

### Backend Approach: Extend Hand-Rolled SigV4 (No New Dependencies)

The existing `generateUploadPresignedUrl()` at `upload.php:177-232` already implements correct SigV4 signing. The new endpoints extend this approach:

| Operation | HTTP Method | Presigned? | Implementation |
|-----------|-------------|------------|----------------|
| `CreateMultipartUpload` | POST to S3 | No — server calls S3 directly | `wp_remote_post()` with SigV4 headers |
| `UploadPart` (presign) | PUT from browser | Yes — presigned URL | Extend existing `generateUploadPresignedUrl()` with `uploadId` + `partNumber` query params |
| `CompleteMultipartUpload` | POST to S3 | No — server calls S3 directly | `wp_remote_post()` with SigV4 headers + XML body |
| `AbortMultipartUpload` | DELETE to S3 | No — server calls S3 directly | `wp_remote_request()` with SigV4 headers |

**Why not the AWS PHP SDK?** The existing SDK at `wordpress/wp-content/plugins/wp-s3-smart-upload/trunk/lib/aws3/` is vendor-prefixed under `BWPS\SSU\Aws3\Aws\S3\S3Client` — not the standard `Aws\S3\S3Client` namespace. Using it couples the theme to a third-party plugin's internal implementation details. Adding Composer to the theme is the right long-term move but is a separate concern. The hand-rolled approach adds ~100-150 lines of PHP with zero new dependencies.

**Single signing mechanism:** All endpoints (existing and new) use the same hand-rolled SigV4 signing. No mixed approaches in the same file.

### CORS Configuration (Wasabi Bucket) — PREREQUISITE

**CRITICAL:** The browser must read the `ETag` response header from each UploadPart response. S3 CORS does not expose `ETag` by default. Without this, the frontend can't collect ETags and `CompleteMultipartUpload` will fail.

**Required CORS rule on the Wasabi bucket:**
```xml
<CORSConfiguration>
  <CORSRule>
    <AllowedOrigin>https://construkted.com</AllowedOrigin>
    <AllowedMethod>PUT</AllowedMethod>
    <AllowedHeader>*</AllowedHeader>
    <ExposeHeader>ETag</ExposeHeader>
  </CORSRule>
</CORSConfiguration>
```

**Defensive code — fail fast on part 1:**
```typescript
// After first part upload completes:
const etag = xhr.getResponseHeader('ETag');
if (!etag) {
    // Abort immediately — don't upload remaining N-1 parts for nothing
    throw new Error(
        'S3 did not return an ETag header. ' +
        'The bucket CORS configuration is likely missing "ExposeHeader: ETag". ' +
        'Contact support.'
    );
}
```

This turns a confusing delayed failure into an immediate, actionable error on the very first part.

**Action:** Verify current CORS config via Wasabi console or CLI before implementation. This is a prerequisite — no code change will work without it.

### ETag Quoting Gotcha

S3 returns ETags wrapped in double quotes: `"d41d8cd98f00b204e9800998ecf8427e"`. The `CompleteMultipartUpload` XML body needs the ETags **with the quotes included**. If the frontend strips them (common mistake when parsing headers), the complete call fails with `InvalidPart`.

```typescript
// CORRECT — keep quotes as-is from the header
const etag = xhr.getResponseHeader('ETag'); // → "\"abc123\""
parts.push({ partNumber: i, etag: etag });  // Pass through unchanged

// WRONG — stripping quotes will cause InvalidPart error
const etag = xhr.getResponseHeader('ETag').replace(/"/g, ''); // DON'T DO THIS
```

### Server-Side Upload State (Prevent IDOR)

The `uploadId` must be validated for ownership on every request. Without this, any authenticated user could hijack or abort another user's multipart upload.

**Store a WordPress transient on create:**
```php
// On CreateMultipartUpload:
set_transient(
    'tszf_multipart_' . $uploadId,
    [
        'user_id' => get_current_user_id(),
        'key'     => $key,
        'created' => time(),
    ],
    7200 // 2 hours — longer than any realistic upload
);
```

**Validate on every subsequent call (sign-part, complete, abort):**
```php
$state = get_transient('tszf_multipart_' . $uploadId);
if (!$state || $state['user_id'] !== get_current_user_id()) {
    wp_send_json_error('Invalid or unauthorized upload', 403);
    exit;
}
```

This also means the S3 key is stored server-side and not trusted from the client — preventing path traversal via tampered key values.

### Security on New Endpoints

All 4 new endpoints must:
- Call `check_ajax_referer('tszf_nonce', 'nonce')`
- Sanitize all inputs (`intval()` for partNumber, validate uploadId format)
- Only register on `wp_ajax_` (authenticated users only)
- Validate uploadId ownership via transient (see above)

**Note:** The existing `get_s3_presigned_upload_url` endpoint has pre-existing security issues (no nonce verification, no input sanitization, registered on `nopriv`). These should be fixed in a **separate PR** to keep this change focused and reviewable.

### 4 Endpoints

| Endpoint | AJAX Action | Purpose |
|----------|-------------|---------|
| Create | `tszf_create_multipart_upload` | Initiate multipart, store transient, return uploadId |
| Sign Part | `tszf_get_upload_part_url` | Presigned URL for one part |
| Complete | `tszf_complete_multipart_upload` | Finalize upload with ETags, delete transient |
| Abort | `tszf_abort_multipart_upload` | Clean up on explicit cancel, delete transient |

The abort endpoint handles explicit user cancellation (clicking the remove button). For implicit abandonment (tab close, browser crash), a **Wasabi bucket lifecycle policy** auto-expires incomplete multipart uploads after 7 days.

### Progress Reporting

Current: Single `xhr.upload.onprogress` gives `loaded/total` for the whole file.

Multipart (sequential uploads): Progress is simpler than with parallel uploads:

```typescript
// Track completed bytes + current chunk progress
const completedBytes = completedParts * PART_SIZE;
const totalProgress = (completedBytes + currentChunkLoaded) / totalFileSize;
```

The existing `_onUploadProgress()` method (lines 375-431) needs modification to accept an offset (bytes already completed from prior parts) so the progress bar, speed, and ETA calculations work correctly across the full file.

### Cancellation Flow

1. User clicks remove button during multipart upload
2. Abort the current in-flight XHR (single XHR since sequential uploads)
3. POST to `tszf_abort_multipart_upload` with the uploadId
4. Server calls `AbortMultipartUpload` on S3, deletes transient

**Tab close / browser crash:** No `sendBeacon` attempt. The Wasabi lifecycle policy handles cleanup. Incomplete parts expire after 7 days. This is the same outcome as a network drop mid-upload.

**Safety net:** Configure a Wasabi bucket lifecycle policy to auto-expire incomplete multipart uploads after 7 days. This is mandatory infrastructure setup.

### Content-Type on UploadPart

Individual `UploadPart` requests should NOT set `Content-Type`. The content type is set on the `CreateMultipartUpload` call. If the presigned URL for `UploadPart` includes `Content-Type` in the signature but the browser sends a different one, the signature will be rejected. Only sign the `host` header for part URLs.

### Wasabi-Specific Testing

Wasabi claims full S3 compatibility but has known quirks with multipart uploads:
- ETag format may differ from AWS S3
- `ListParts` pagination behavior can differ
- Some regions have had delayed cleanup after `AbortMultipartUpload`

**All multipart operations must be tested against actual Wasabi, not assumed to work from AWS S3 docs.**

### Deployment

**Order: PHP first, then JS.** This is safe because the old JS never calls the new endpoints — deploying PHP first is a no-op. When the new JS deploys, the endpoints are already available.

**Manual deployment steps:**
1. Deploy PHP changes to `construkted_reality_v1.x` (theme)
2. `cd construkted.uploadjs && npm run build`
3. Copy `dist/construkted.upload.js` → `construkted_reality_v1.x/wp-content/themes/gowatch-child/includes/frontend-submission/assets/js/construkted.upload.js`
4. Commit in both repos
5. Deploy WordPress theme

## System-Wide Impact

- **Interaction graph:** Upload flow is self-contained: frontend-form.ts → TSZF_UPloader.ts → upload.php → Wasabi. No callbacks, no observers, no middleware. The `afterUploadCallback` in `startUploading()` fires after each file completes — this stays unchanged.
- **Error propagation:** New failure modes for >5GB path: individual part failure (surface error to user), CreateMultipartUpload failure (surface error), CompleteMultipartUpload failure (retry up to 3 times automatically before surfacing error — all parts are already uploaded so retrying is cheap).
- **State lifecycle risks:** Incomplete multipart uploads consume storage on Wasabi until explicitly aborted or expired (7 days). Lifecycle policy is the primary cleanup mechanism. Transients auto-expire after 2 hours.
- **API surface parity:** No other interfaces use this upload path. The `construkted_api` Node.js backend uses `aws s3 cp` CLI for its own S3 operations — completely separate.
- **Regression risk:** Minimal. The existing <5GB upload path is completely untouched. Only files >5GB use the new code.

## Dependencies & Risks

| Risk | Impact | Mitigation |
|------|--------|------------|
| Wasabi CORS not exposing ETag | Multipart uploads fail at completion | Defensive check on part 1 fails fast with clear error. Verify CORS before any code changes. |
| Wasabi multipart behavior differs from AWS | Parts upload but complete/abort behaves unexpectedly | Test all operations against actual Wasabi, not just assumed from AWS docs |
| Version skew between JS and PHP deploys | New JS calls endpoints that don't exist yet | Deploy PHP first, then JS. Old JS never calls new endpoints. |
| Hand-rolled SigV4 bugs in new operations | CreateMultipartUpload or CompleteMultipartUpload signing fails | The existing presigned URL signing is proven correct. New operations use the same signing logic. Test thoroughly. |
| Transient expiry during very long upload | uploadId validation fails mid-upload | 2-hour transient TTL. A 50GB file at 10MB/s takes ~85 minutes. Tight but sufficient. Monitor and increase if needed. |
