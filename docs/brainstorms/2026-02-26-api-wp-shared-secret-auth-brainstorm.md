# Replace IP Allowlist with Shared Secret Authentication (API to WP)

## Problem

The `construkted_api` processing servers call back to the WordPress site's `control_asset_details.php` endpoint after asset processing completes (to update custom fields, report errors, etc.). The only protection on this endpoint is a **hardcoded IP allowlist**.

The processing servers (2-3 machines) run on **residential internet connections with dynamic IPs**. When an IP changes (every few months), the endpoint silently rejects requests. The failure is only discovered when end users report that their assets aren't processing — there's no monitoring or alerting.

**Current state:**
- API -> WP callbacks: `axios.get()` with zero auth headers
- WP protection: hardcoded `$allowedIPs` array in `control_asset_details.php`
- No shared secrets, tokens, or API keys between the two systems
- WP admin has a "Backend Server URLs" tab for configuring API server addresses

## What We're Building

Replace the hardcoded IP allowlist in `control_asset_details.php` with a **shared secret (API key)** that the processing servers include as an HTTP header in every callback to WordPress.

**How it works:**
1. A shared secret is generated and stored in two places:
   - WP admin settings (alongside the existing "Backend Server URLs" configuration)
   - Each API server's `.env` file
2. When the API server calls back to WP, it includes the secret as an `X-Construkted-Key` header
3. WP's `control_asset_details.php` reads the header and validates it against the stored secret
4. If the key matches, the request proceeds. If not, it's rejected with a 403.
5. The hardcoded `$allowedIPs` array is removed entirely

This completely eliminates the dependency on IP addresses. Server IPs can change freely without breaking processing callbacks.

## Why This Approach

**Approaches considered:**

| Approach | Verdict |
|----------|---------|
| **A. Simple shared API key** | **Chosen** - Simplest solution, eliminates the IP problem entirely |
| B. Per-server API keys | Rejected - Unnecessary complexity for 2-3 servers. No need for per-server revocation. |
| C. HMAC request signing | Rejected - Overkill. HTTPS already encrypts the key in transit. |
| D. Dynamic IP registration | Rejected - Still relies on IP-based auth, just automates the updates. More complex and fragile. |

**Rationale:** The IP allowlist was always a proxy for "is this request from a trusted server?" A shared secret answers that question directly without IP dependency. With HTTPS encrypting all traffic, sending the key as a header is secure. YAGNI rules out the more complex alternatives.

## Key Decisions

- **Single shared key for all servers**: One secret shared across all 2-3 processing servers. Simple to manage, and there's no current need for per-server granularity.
- **Header-based authentication**: The key is sent as an `X-Construkted-Key` HTTP header rather than as a query parameter (headers aren't logged in server access logs by default, query params are).
- **WP admin for key management**: The shared secret is configured on the **"Backend Server URLs" tab** in the existing WP admin settings page (`Settings > Construkted`), keeping it alongside the server URL configuration it relates to.
- **API server `.env` for key storage**: The key is stored in the API server's `.env` file, consistent with how other secrets (S3 keys, Cesium token) are already managed.
- **Remove IP allowlist entirely**: The IP check is removed rather than kept as a secondary layer. Maintaining it alongside the shared secret adds complexity for no real benefit (the key is the auth, the IP is just noise).
- **Auto-generated key with regenerate button**: WP generates the key (secure random string), displayed in admin with a copy button and a "Regenerate" button for key rotation. Admin copies the key to each API server's `.env`.

## Resolved Questions

- **Where in WP admin should the key field go?** -> **"Backend Server URLs" tab** — keeps the shared secret alongside the server URL configuration it directly relates to.
- **Should the key be auto-generated by WP or manually set?** -> **Auto-generated with a "Regenerate" button** — WP generates a secure random key, admin copies it to each API server's `.env` file. Regenerate button allows key rotation.

## Security Concerns (Out of Scope - Document Separately)

The following security issues were identified during research but are **not addressed by this feature**. They should be captured in a separate document for future planning:

1. **WP -> API direction is completely unprotected**: The WP site calls the API's `/request_tiling` endpoint via plain `curl_init()` with no authentication. Anyone who discovers the API endpoint can submit arbitrary tiling tasks.
2. **API server has no auth middleware**: The API server has zero authentication on any endpoint and uses `Access-Control-Allow-Origin: *`.
3. **`control_asset_details.php` uses `$_REQUEST` without CSRF protection**: Request parameters are read directly from `$_REQUEST` with minimal sanitization.
4. **`wPServerUrl` is passed as a query parameter**: The WP URL is sent to the API as part of the tiling request, meaning a malicious actor could redirect callbacks to a different server.
